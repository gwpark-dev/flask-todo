# add-route — 새 Flask 라우트 추가 절차

이 Skill 은 `flask-todo` 프로젝트에 새 라우트를 추가할 때 따라야 하는 **표준 절차**입니다.
프로젝트 성격 자체가 학습용 튜토리얼이라, 빠르게 동작하는 코드보다 **"왜 이렇게 쓰는지"** 가 길게 풀려 있는 코드가 더 가치 있습니다. 이 Skill 은 그 톤을 새 라우트에서도 자동으로 유지시키는 역할을 합니다.

> 원본 규칙은 저장소 최상단 `CLAUDE.md` 에 있습니다. 이 문서는 그 규칙들을 "라우트 추가" 라는 한 작업 단위로 묶어 둔 체크리스트입니다.

---

## 1) 언제 이 Skill 을 쓰는가

다음과 같은 사용자 요청이 들어오면 **반드시** 이 Skill 의 절차를 따라 진행합니다.

- "새 라우트 추가해 줘"
- "/foo 엔드포인트 만들어 줘"
- "버튼 누르면 ~ 하는 기능 추가해 줘"
- "POST 라우트 추가해 줘", "GET 라우트 만들어 줘"
- "할 일을 ~ 하는 동작 만들어 줘" (예: 일괄 완료, 정렬 변경 등)

반대로, 다음 작업은 **이 Skill 의 대상이 아닙니다**(별도 작업으로 다루세요).

- 기존 라우트(`add` / `toggle` / `delete` / `index`) 의 로직만 수정 — 패턴은 이미 잡혀 있음.
- `Todo` 모델의 컬럼만 추가/변경 (라우트는 그대로) — 스키마 마이그레이션 작업.
- `templates/index.html` 의 디자인/CSS 만 손보는 작업.

## 2) 사전 점검 (Phase 0 — 코드 쓰기 전에 멈출 것)

코드를 손대기 전에 다음 세 가지를 먼저 확인합니다.

1. **정말 새 라우트가 필요한가?** 기존 4개 라우트를 조합해서는 처리 불가능한가? (CLAUDE.md 의 YAGNI 원칙)
2. **모델(`Todo`) 컬럼 변경이 동반되는가?**
   - YES → 사용자에게 먼저 알립니다. 정확히 다음과 같이:
     > "스키마 변경이 동반됩니다. `db.create_all()` 은 이미 존재하는 테이블을 변경해 주지 않으므로, `instance/todo.db` 를 삭제해서 다시 만들거나 Flask-Migrate 를 도입해야 합니다. 어느 쪽으로 가시겠어요?"
   - 사용자 합의 전에는 컬럼 정의를 만지지 않습니다.
   - NO → 다음 단계.
3. **변경이 비자명하면 계획부터** 보여주고 사용자 동의를 받습니다 (CLAUDE.md "프로젝트 성격" 절). 라우트 1개 + 폼 1개로 끝나는 단순 추가는 곧장 진행해도 됩니다.

## 3) 절차 체크리스트 (5단계)

### 3-1. `app.py` 에 라우트 함수 추가
- 위치: 기존 라우트(`add` / `toggle` / `delete`) 묶음 **뒤**, 부트스트랩 블록(`with app.app_context():`) **위**.
- 부트스트랩 블록을 `if __name__ == "__main__":` 안으로 옮기지 마세요. import 만 해도 테이블/시드가 준비되어야 하므로 분리 상태를 유지합니다.

### 3-2. PRG 패턴 준수 (변경 라우트의 경우)
- POST 라우트의 마지막 줄은 **반드시** `return redirect(url_for("index"))`.
- 새로고침 시 같은 POST 가 다시 날아가는 사고를 막기 위함입니다 (예: 같은 항목이 두 번 추가되는 사고).
- GET 라우트라면 `render_template(...)` 또는 다른 응답으로 끝내면 됩니다.

### 3-3. `templates/index.html` 에 폼 추가 (사용자가 트리거할 입구가 필요할 때)
- `action="{{ url_for('함수명', 인자=값) }}"` 형식 강제. 절대 `/toggle/3` 같은 경로를 하드코딩하지 마세요 (학습 포인트 중 하나).
- 변경 라우트의 폼은 `method="post"`. GET 으로 보내면 URL 에 입력값이 노출되고 새로고침 사고도 잘 납니다.
- 입력값을 받는 폼이라면 `<input name="키">` 와 서버의 `request.form.get("키")` 의 키 이름이 정확히 일치해야 합니다 (가장 흔한 함정).

### 3-4. 튜토리얼 주석 작성
- 기존 라우트들(`app.py:154-227`)과 **같은 주석 밀도**를 유지합니다.
- "왜 이렇게 쓰는지" 를 길게 풀어 씁니다. 한 줄짜리 단순 설명 주석은 쓰지 않습니다.
- 한국어로 씁니다.
- 새 학습 개념이 들어오면 학습 포인트 라벨을 부여합니다 (아래 §5 참고).

### 3-5. 검증
1. `python app.py` 로 서버 실행 (debug 모드라 코드 저장 시 자동 재시작).
2. 브라우저로 `http://127.0.0.1:5000` 접속.
3. 새 동작이 의도대로 작동하는지 확인.
4. 변경 라우트의 경우, **새로고침을 한 번** 눌러서 같은 동작이 중복 실행되지 않는지(=PRG 정상 작동) 확인.
5. 학습 포인트 주석이 의도대로 보이는지 한번 더 읽어 봅니다.

## 4) 라우트 함수 템플릿

### 변경 라우트 (PRG 패턴) — 가장 자주 쓰는 골격
```python
@app.route("/엔드포인트/<int:todo_id>", methods=["POST"])
def 함수명(todo_id):
    # PK 로 단건 조회 — 없으면 None.
    # SQL 로 치면 SELECT * FROM todo WHERE id = ? LIMIT 1.
    todo = Todo.query.get(todo_id)

    # 존재할 때만 변경. None 가드를 빼먹으면 AttributeError 가 납니다.
    if todo is not None:
        # 변경 로직: 속성 대입 / db.session.delete / 등등.
        # 객체 속성을 바꿨다고 자동으로 DB 가 업데이트되지는 않습니다.
        # 세션에 "이 객체가 바뀌었다" 흔적만 남고, commit() 시점에 UPDATE 가 날아갑니다.
        db.session.commit()

    # PRG 마무리: 새로고침 사고 방지를 위해 반드시 redirect 로 끝냄.
    return redirect(url_for("index"))
```

### 폼 입력을 받는 변경 라우트 (`add()` 형태)
```python
@app.route("/엔드포인트", methods=["POST"])
def 함수명():
    # request.form 은 폼 데이터를 담은 "딕셔너리 비슷한" 객체.
    # .get("키", "") 로 키가 없을 때도 KeyError 안 나게.
    # .strip() 으로 공백만 입력한 경우도 빈값으로 취급.
    값 = request.form.get("키", "").strip()

    # 빈 입력은 조용히 무시 (저장하지 않고 페이지만 새로 그림).
    if 값:
        # 객체 생성/조회/변경 로직.
        db.session.commit()

    return redirect(url_for("index"))
```

### 조회 라우트 (GET)
```python
@app.route("/경로")
def 함수명():
    # 필요한 데이터를 미리 계산해서 템플릿에 넘김.
    # 표시용 계산은 가능하면 파이썬 쪽에서 미리 끝내고, 템플릿은 "출력만" 맡깁니다.
    데이터 = Todo.query.filter_by(...).all()
    return render_template("index.html", 데이터=데이터, ...)
```

## 5) 학습 포인트 라벨링 규칙

기존 `app.py` 에는 학습 포인트가 세 곳에 라벨링돼 있습니다.

| 라벨 | 위치 | 주제 |
| --- | --- | --- |
| **[A]** | `add()` 라우트 안 | `db.session.add()` 와 `db.session.commit()` 의 관계 (장바구니 → 결제) |
| **[B]** | `Todo` 모델 안 | `query.all()` / `query.get()` / `query.filter_by()` 세 메서드의 차이 |
| **[C]** | `Todo` 모델 위 | ORM 이 SQL 을 어떻게 대체하는가 — 대비표 |

새 라우트가 **새 학습 개념**을 도입한다면 다음 알파벳을 부여합니다:

- 다음 사용 가능 라벨: **[D]** → **[E]** → **[F]** → ...
- 새 학습 개념의 예: 폼 검증, `flash()` 메시지, JSON 응답(`jsonify`), 트랜잭션 롤백(`db.session.rollback()`), 페이지네이션, 정렬/필터링 쿼리 등.

새 라우트가 **기존 개념의 재활용**이면 라벨을 부여하지 않습니다. 대신 본문에서 짧게 가리킵니다:

> `# (→ [학습 포인트 A] 참고: add 와 동일한 session.add/commit 패턴)`

라벨 박스 스타일은 기존 `app.py:86-111` 의 양식을 그대로 따라 씁니다 — 위아래 `─` 구분선, 라벨명 한 줄, 본문 여러 줄, 마지막 `─` 구분선.

## 6) 함정 체크리스트 (작업 끝나기 전 자체 점검)

작업을 마치기 전, 다음 항목을 한 줄씩 확인하세요.

- [ ] 동적 파라미터에 **`int:`** 컨버터를 붙였는가? `<todo_id>` 만 쓰면 문자열로 들어와 `Todo.query.get(...)` 결과가 항상 `None` 이 되는 함정이 있습니다.
- [ ] POST 라우트의 끝줄이 **`return redirect(url_for("index"))`** 인가? (PRG)
- [ ] DB 변경 후 **`db.session.commit()`** 을 호출했는가? 누락 시 서버 재시작 때 입력이 통째로 사라집니다.
- [ ] **`Todo.query.get(...)`** 형태를 유지했는가? SQLAlchemy 2.0 에서는 `db.session.get(Todo, ...)` 가 권장이지만, 학습 자료가 거의 다 `.query.get` 형태라 일부러 그대로 둡니다. 사용자가 "이제 마이그레이션 해 보자" 라고 **명시**하기 전에는 바꾸지 마세요.
- [ ] 템플릿/리다이렉트에 **`url_for()`** 만 썼는가? `/toggle/3` 같은 경로 하드코딩 금지.
- [ ] 모델 컬럼을 바꾼 경우, **`instance/todo.db` 처리**(삭제 또는 Flask-Migrate 도입)에 대해 사용자 합의를 받았는가? `db.create_all()` 은 기존 테이블을 변경하지 않습니다.
- [ ] **부트스트랩 블록**(`with app.app_context():` + `db.create_all()` + 시드 삽입)을 건드리지 않았는가? import 만 해도 테이블/시드가 준비되어야 하므로 모듈 최상단 유지.
- [ ] **`instance/`, `*.db`, `venv/`** 가 커밋 대상에 들어가지 않았는가? 이미 `.gitignore` 에 등록돼 있지만, 새 폴더를 만들었다면 한 번 더 확인.
- [ ] 새 라우트의 주석 밀도가 기존 라우트와 **같은 수준**인가? 한 줄 요약 주석으로 끝내지 않았는가?
- [ ] 헬퍼 함수/유틸로 추상화하고 싶은 욕구를 참았는가? (CLAUDE.md "수정 작업 톤" — 비슷한 코드 3줄이 반복돼도 빼지 않는 편이 학습에 도움.)

## 7) HTML 폼 작성 패턴

### 단순 트리거 폼 (파라미터 없는 POST — 예: "전체 완료" 버튼)
```html
<form action="{{ url_for('함수명') }}" method="post">
    <button type="submit">버튼 텍스트</button>
</form>
```

### 동적 파라미터 폼 (특정 todo 대상 — 예: 토글/삭제)
```html
<form action="{{ url_for('함수명', todo_id=todo.id) }}" method="post">
    <button type="submit">…</button>
</form>
```
- `url_for('함수명', todo_id=todo.id)` 가 자동으로 `/엔드포인트/3` 같은 경로를 만들어 줍니다.
- 라우트 함수 데코레이터의 `<int:todo_id>` 와 키워드 인자 이름(`todo_id`)이 정확히 일치해야 합니다.

### 입력값을 받는 폼
```html
<form action="{{ url_for('함수명') }}" method="post">
    <input type="text" name="키" placeholder="...">
    <button type="submit">제출</button>
</form>
```
- `<input name="키">` 의 `name` 값과 서버 `request.form.get("키")` 의 키 이름이 **정확히 일치**해야 합니다. 한 글자만 어긋나도 입력값이 통째로 사라집니다 — 가장 흔한 함정.

### 접근성
- 아이콘만 있는 버튼(예: ×, ✓ 등)은 반드시 **`aria-label="..."`** 을 부여합니다. `templates/index.html:232` 의 삭제 버튼이 좋은 예시입니다.
  ```html
  <button type="submit" class="delete-btn" aria-label="삭제">×</button>
  ```
  화면낭독기(스크린리더) 사용자는 시각 정보 없이 듣기만 하므로, `×` 같은 기호 하나만 있으면 동작을 알 수 없습니다.

## 8) 마무리 톤

- 사용자는 **한국어**로 소통합니다. 응답/계획/주석 모두 한국어로.
- 새 기능을 추가할 때도 기존 라우트들과 **같은 튜토리얼 주석 밀도**를 유지합니다.
- 추상화 욕구가 들어도 참습니다. 한 파일/한 함수에서 비슷한 코드 3줄이 반복돼도 헬퍼로 빼지 않는 편이 학습에 더 도움이 됩니다.
- 비자명한 변경은 계획부터 보여주고 사용자 동의를 받은 뒤 구현합니다.
